## Context

`@skillet-cli/core`'s `run()` function accepts an optional `skillDir` argument. When provided, it is used as the absolute path to the skill directory. When absent, `run()` calls `readSkilletMarker()` which reads `skillet.skills` from `package.json` — a *parent* directory whose subdirectories are scanned for `SKILL.md` files.

`create-skillet` generates a `bin/cli.js` that always passes `skillDir` as an absolute URL-resolved path, so the `package.json` fallback is never reached at runtime. The `skillet.skillDir` field that `create-skillet` writes to `package.json` is thus dead config — present but never consulted.

The two keys have different semantics:
- `skillet.skills` — a parent directory; skill trees are its *subdirectories* with `SKILL.md`
- `skillet.skillDir` — a direct path; the skill tree *is* this directory (contains `SKILL.md` itself)

## Goals / Non-Goals

**Goals:**
- `package.json`'s `skillet.skillDir` is the runtime source of truth for single-skill packages generated by `create-skillet`
- Generated `bin/cli.js` contains no path information — it is stable across file-move operations
- Core's `run()` resolves skill location from `package.json` when `skillDir` is not passed programmatically
- Post-move step only updates `package.json`; no binary rewrite needed

**Non-Goals:**
- Changing the `skillet.skills` parent-directory discovery mechanism (multi-skill packages remain unaffected)
- Changing the programmatic `run({ skillDir })` API (stays supported for custom wrappers)
- Migrating existing published packages (existing `bin/cli.js` files continue to work as-is)

## Decisions

### Decision 1: Add `skillDir` to `SkilletPackageJson` and `SkilletMarker`

**Chosen**: Extend `readSkilletMarker` to recognize `skillet.skillDir` as a direct skill path. Return it via a new `directSkillDir` field on `SkilletMarker`. In `run()`, when `marker.directSkillDir` is set, use it directly without calling `discoverSkillTrees`.

**Alternative considered**: Change `create-skillet` to write `skillet.skills` instead of `skillet.skillDir`. Rejected because `skillet.skills` semantics require a *parent* directory whose *subdirectories* contain `SKILL.md`. For a skill at `skill/SKILL.md`, the parent would be `.` (the package root), and `discoverSkillTrees(".")` would scan all subdirectories — including `node_modules/`, `bin/`, etc. — which is fragile and expensive.

**Alternative considered**: Resolve `skillDir` relative to `import.meta.url` in the generated binary but read it from `package.json` at generation time instead of hardcoding. Rejected because this still makes the binary stale after a move — the same root cause.

### Decision 2: Precedence — programmatic `skillDir` overrides `package.json`

Programmatic callers who pass `skillDir` to `run()` retain their current behavior. The new `directSkillDir` path is only taken when `skillDir` is `undefined`. This preserves backward compatibility for custom wrappers.

### Decision 3: Resolution — `skillet.skillDir` resolved relative to `process.cwd()`

`readSkilletMarker` receives `packageRoot` (always `process.cwd()` from `run()`). The relative `skillDir` value is resolved with `path.resolve(packageRoot, skilletObj.skillDir)`. Since `npx . install` sets `cwd` to the package directory, this is always correct.

### Decision 4: `skillDir` takes precedence over `skills` when both are present

If a `package.json` contains both `skillet.skillDir` and `skillet.skills`, `skillDir` is used. This avoids ambiguity and matches the single-source-of-truth intent. A warning is not emitted (the combination is unlikely in practice).

### Decision 5: `buildBinCliJs` becomes zero-argument

The `skillDir` parameter is removed from `buildBinCliJs()`. The generated output is now a fixed template — no runtime path interpolation. This also removes the only use of `fileURLToPath`/`new URL()` in the generated file, simplifying it.

## Risks / Trade-offs

**Risk**: Existing published packages have a hardcoded `skillDir` in `bin/cli.js` and no `skillet.skillDir` in `package.json`.
→ No regression. Those packages still pass `skillDir` to `run()` directly; the new path is only taken when `skillDir` is `undefined`. The `skillet.skillDir` key was already being written by `create-skillet` (via `npm pkg set`), so packages generated after the bugfix release will have it.

**Risk**: A user manually edits `skillet.skillDir` in `package.json` to a wrong value — installs fail silently.
→ The error from `normalizeSkill()` will surface a clear message about missing `SKILL.md`. This is no worse than the current state where the binary path is wrong.

**Trade-off**: The generated `bin/cli.js` now depends on `process.cwd()` being the package root at runtime. This is guaranteed by `npx .` but not by direct `node bin/cli.js` invocations from a different directory.
→ Acceptable: `npx .` is the documented invocation. Direct invocations with wrong cwd are unsupported.

## Migration Plan

- No migration required for consumers of `@skillet-cli/core` using the programmatic API.
- New `bin/cli.js` template takes effect for all packages generated after this release.
- Existing generated packages continue to work unchanged (backward-compatible).
- No rollback complexity — changes are additive to `core`; changes to `create` only affect newly generated packages.

## Open Questions

None — all design decisions are resolved.
